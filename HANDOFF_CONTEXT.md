## 2. Что такое AnchorVault (одним абзацем)

Мульти-ассет on-chain хранилище для ERC-20. Главная идея — авторизация
операций через EIP-712 подписи отдельных ключей сейфа
(`mainAuthKey` / `recoveryAuthKey`), а **не** через приватник кошелька
владельца. Фишинг-устойчивость: даже если владелец подписывает
зловредную транзакцию своим обычным ключом, сейф нельзя опустошить —
для этого нужны подписи вспомогательных ключей, которые хранятся
отдельно. Дополнительно: таймлоки, добровольная блокировка, ролевое
разделение (creator/guardian), пауза, фиксированный «холодный»
адрес для panic-вывода.

## 8. Архитектура — самое важное, что надо знать сразу

### Роли
- **Owner** (по сейфу) — это EOA, к которой сейф привязан. **Не имеет
  права подписывать операции сейфа напрямую.**
- **mainAuthKey** (по сейфу) — ключ для обычных операций (withdraw,
  deposit, transfer, и т.д.).
- **recoveryAuthKey** (по сейфу) — ключ для экстренных операций
  (earlyClose, recoverToSafe, emergencyWithdrawToAny).
- **Creator** (глобально) — деплоер. Управляет supported tokens, welcome
  bonus, вывод пулов комиссий с 7-дн таймлоком.
- **Guardian** (глобально) — может только ставить паузу. Не выводит
  деньги, не меняет роли. Кулдаун 2 дня обычно, мгновенно — в emergency.
- **globalEmergency[owner]** — куда летит panicWithdraw. Меняется через
  7-дн таймлок. На каждого владельца — свой.

### EIP-712 авторизация (суть)
Каждая операция — это типизированное сообщение (тип-хэш типа
`WITHDRAW_TYPEHASH`, `EARLY_CLOSE_TYPEHASH`, …) с параметрами:
- `vaultId`, `nonce` (per-vault), `deadline`, плюс параметры операции.
- Доменный сепаратор: chainId + address(this) + версия.

Контракт извлекает `ecrecover(...)` и сравнивает с
`v.mainAuthKey` / `v.recoveryAuthKey`. Совпало → инкрементирует
`nonce`, выполняет операцию. Не совпало → revert.

Replay-защита: nonce монотонно растёт; deadline ограничивает окно;
domain separator привязывает к конкретному контракту и сети.

### Уровни сейфа и комиссии
| Уровень   | Deposit fee | Max timelock |
| --------- | ----------- | ------------ |
| SAFE      | 0.5%        | 0 (нет)      |
| VAULT     | 1.5%        | 72ч          |
| FORTRESS  | 2.0%        | 168ч         |

### Иерархия экстренных (от мягкого к крайнему)
| Операция                 | Штраф | Подпись   | Куда деньги            |
| ------------------------ | ----- | --------- | ---------------------- |
| withdraw                 | 0.5%  | main      | любой адрес            |
| earlyClose               | 5%    | recovery  | владельцу              |
| recoverToSafe            | 10%   | recovery  | v.emergencyAddress (snapshot) |
| emergencyWithdrawToAny   | 15%   | recovery  | любой адрес            |
| panicWithdraw            | 20%   | НЕТ       | globalEmergency (live) |

## 9. Known issues (4+2)

Полно — в `KNOWN_ISSUES.md`. Кратко:

1. **I-1.** `recoverToSafe` использует snapshot `emergencyAddress`,
   `panicWithdraw` — live. Это **намеренно**.
2. **I-2.** Один `nonce` на main и recovery — pre-signed recovery
   аннулируется любой main-операцией. Принято.
3. **I-3.** Грифинг входящего слота secureTransfer (48ч). Low. Фикс
   планируется в V45.1 после аудита.
4. **I-4.** forge-lint `unsafe-typecast` — false positives.
5. **I-5.** Старый комментарий в коде про «без via_ir» — устарел, фикс.
6. **I-6.** Orphan MockANCR на Sepolia — артефакт деплоя, не влияет.
